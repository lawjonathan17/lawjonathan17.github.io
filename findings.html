USER RESEARCH

User Interviews

View the user interviews

User interviews are important in the process of understanding the users of the product and their motives. The developer can draw information from user interviews about what type of content the audience wants to see and how to present them. Some of the questions I asked were getting to know the user questions like "What is your current occupation?", etc. To get a sense of what content to put on my portfolio website, I asked, "What is the most important feature for the portfolio?" and "If you were to create your own portfolio website, what are some features you think would add value to it?"

Competitor Analysis

View the competitor analysis

A competitor analysis is a review of competing or similar web pages and finding their strengths and weaknesses. I reviewed other Computer Science students, UX designers, freelance web developers and my own LinkedIn account. I reviewed other students' webpages to find out what content is being presented by my immediate competitors for the jobs that I am trying to get. I researched UX designers to see what a more experienced web developer would add to their portfolio website. I reviewed my LinkedIn website to get key features from a popular professional networking website.

Feature Value Matrix

View the feature value matrix

A feature value matrix is an analysis of features by rating them based on their user or business value and how much effort it will take to complete each task. A feature value matrix is used to order tasks; where the goal is to find high value, low effort tasks to complete first. Some of the features that I selected were the requirements of midterm part 2, where they had the highest user value. Other features that I added were from looking at my competitors' portfolio websites and drawing features that I liked and wanted to include in my own. These features include: a link to my resume, GitHub, and a way to contact me by email.

Feature Prioritization

View the feature prioritization

I prioritized my features in the "Feature Prioritization chart" by low-priority (left) to increasing priority (right). For the Kano model, I prioritized the features based on their value and effort levels conducted from the feature value matrix. While building my interface I started from the tasks that had the most value and the least amount of effort then made my way to tasks that had high value and more effort. Then I did the lowest value and high effort tasks last. I referenced the feature prioritization chart a lot and went from right to left. Using these charts while developing the interface of my site helped me manage my time and ensure that I had the most important features completed.

USER TESTING

User Stories

View the user stories

User stories are used to capture the perspective of the end user with the product. User stories help find out what the user wants and how it is achieved with the product. Most of my user stories were from the perspective of a Software Engineering Hiring Manager because that is the ideal audience of my website. I created the user stories by thinking as if I were a Software Engineering Hiring Manager and thinking about what features they look for and how to achieve them.

Cognitive Walkthrough

View the cognitive walkthrough

A cognitive walkthrough is where a user works through a series of tasks and is asked a set of questions. A cognitive walkthrough is important to understand the usability and learnability of the product by asking a user questions while they are navigating through the website. Cognitive walkthroughs can provide useful information for what the user likes about the product, to use more of, or what the user dislikes about the website, to improve on.

SUMMARY OF FINDINGS

Before I started the implementation of my portfolio website, I had to gather requirements based on what my target audience would want to see. From the user interviews, I was able to conclude that projects and experience were a focus area for the content of my portfolio website. From the competitor analysis, I was able to retrieve some features that I liked and wanted to implement into my own website including: fixed navigation bar, a link to my GitHub and LinkedIn. Prioritizing the tasks from the feature value matrix and Kano model helped me organize the development process so that the most important tasks were completed first.

After the implementation, I was able to get feedback for the product that I built. Some of the feedback that I got from the cognitive walkthrough was to add more ways to add style to my website. My website looked a bit bland and needed some more of my personality in it. I also learned that I could add a skills section showing which technical skills that I know. The cognitive walkthrough also let me know that I had some spacing issues in my website that distracted the user's attention. The user tests gave me information that my users liked my redesign a lot more than the original design and some certain aspects of it that looked nicer, such as the background image and nicer font.

Some challenges that I faced while gathering these findings were thinking in the perspective of the user reviewing my product. It was challenging in that I had to use a new way of thinking to imagine that I was a hiring manager or another software engineer and I had to think about what that user would like. Another challenge was rating the values of features for the feature value matrix in terms of the user and business. Once again, I had to try to imagine what the user would want and try to think of what they would find valuable.

My findings played a key role in my entire development process. I was able to find out what content to put from the user interviews and competitor analysis.
